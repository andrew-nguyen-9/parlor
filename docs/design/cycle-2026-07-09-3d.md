## Invariants every 3D/mechanic unit inherits

1. **Logic core stays deterministic ‚Äî 3D is presentation only.** Win = a logic gate,
   never a rigid-body simulation. Puzzle logic is pure + date-seeded (`lib/rng.ts`
   `daySeed`); the generator guarantees a UNIQUE solution; NO solve-time RNG. Three.js
   renders the solved/solving state; trivia is flavor on top. Kinematic motion only ‚Äî
   **no physics engine** (guarantees unique-solve + mobile perf).

2. **Generator signature is frozen:** `generate<X>(dayIndex: number, date: string)`
   ‚Äî **`date` is a `string`, not a `Date`** (verified in code 2026-07-09; the older
   `engine-contracts.md` table says `Date` ‚Äî it is STALE on this one point, trust this).
   A unit may redefine the payload interface internals freely, but the export NAME +
   `(number, string)` signature must not change.

3. **`frontend/lib/queries.ts` is a shared loader monolith ‚Äî NO unit edits it.** It owns
   every `get<X>Puzzle(date?)` loader with the inline offline fallback
   `generate<X>(dayIndexOf(day), day)`. Because the loader imports `generate<X>` by name,
   keeping invariant 2 means queries.ts never needs touching ‚Üí the room units run one
   parallel wave without colliding. If you think you need to edit queries.ts, you have
   broken invariant 2 instead ‚Äî don't.

4. **Offline / zero-env is hard (CLAUDE.md).** New deps bundle at build and add NO runtime
   network dependency. The MAP polygon fallback (`WorldMap.tsx`) and the zero-env seed bank
   must keep working with no network. No CDN / tile / API in the offline path.

5. **Mobile = hard requirement.** Every room fully playable in **portrait**, no rotate
   prompt, **no horizontal scroll on any element at any breakpoint** (test 320 / 768 /
   1440px+). 3D cameras framed for a tall viewport.

## Shared primitives (F1 builds; consumers listed)

### `components/ThreeStage.tsx` ‚Äî shared Three.js canvas wrapper  (consumers: E2, E3)
One client component that owns the boilerplate every 3D room repeats and mobile-perf
traps that are easy to get wrong:
- mounts a `<canvas>`; `WebGLRenderer({ antialias, powerPreference:"high-performance" })`;
  **DPR capped ‚â§2** (mobile perf); `outputColorSpace` sane default.
- takes a `scene`, `camera` (or a factory) + an `onFrame(dt)` callback; runs ONE shared
  `requestAnimationFrame` loop; passes delta time.
- `ResizeObserver` on the container ‚Üí renderer + camera aspect update (fills width,
  no x-overflow); **portrait-first camera framing** helper (fit the target in a tall box).
- **Disposes** geometries / materials / textures / renderer on unmount (leak-free).
- Respects `prefers-reduced-motion` (pause or slow the loop; a designed state).
- API is small + stable; document the props at the top of the file. E2/E3 supply only
  their scene contents + `onFrame`.

### `components/FluidStage.tsx` ‚Äî full-width responsive container  (consumers: E1, E9)
The "no dead L/R gutters, no horizontal scroll" primitive. A container that spans the
available width from smallest‚Üílargest viewport with `clamp()`-based adaptive padding,
never a fixed narrow column, never overflows the x-axis. E1 (S√©ance) and E9 (global)
compose it instead of re-deriving gutter math per room.
